# Supervertaler Workbench v1.9.490

One small follow-up fix: the same Phrase bilingual DOCX import bug from v1.9.487 / v1.9.488, which turned out to have a second copy of the broken detection logic that v1.9.487 missed.

---

## What's fixed

### Phrase bilingual DOCX import still rejected short files via the omnibus detector

A user reported their Phrase-exported bilingual DOCX (`Praxe_PC-cs-de_de-TR.docx`, 63 segments, 8 grid columns) being rejected on import — the same file, with the same warning, that v1.9.487 was supposed to have fixed.

Root cause: there are two Phrase detectors in the code.

- The dedicated `detect_phrase_docx` in `modules/phrase_docx_handler.py` *was* fixed in v1.9.487 to drop the `> 100 rows` threshold and accept 7- or 8-column tables.
- But `import_phrase_bilingual` in `Supervertaler.py` actually calls `detect_bilingual_docx_type` in `modules/trados_docx_handler.py` — the omnibus detector that dispatches Trados / CafeTran / Phrase / memoQ — and *that* one still had the old `len(table.rows) > 100 and cells >= 7` check.

v1.9.487 only updated one of the two paths, so the omnibus detector kept returning `"unknown"` for short Phrase files and the UI kept throwing the warning before the (already-fixed) parser ever got a chance.

Fix aligns the Phrase branch in `detect_bilingual_docx_type` with the corrected logic — `len(rows) >= 2 and len(cells) in (7, 8) and ':' in first_cell`. Single source of truth would be cleaner; consolidating the two detectors can come later.
